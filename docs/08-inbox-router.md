# Inbox路由器规则

Inbox Router 的任务是把捕获内容快速分配到正确去向，并生成下一步行动。它不是新的复杂系统，而是减少手动分类摩擦的前置处理器。

## 处理顺序

每条 Inbox 内容按下面顺序判断：

1. 是否明显无价值、重复、过期？是则建议 `discard`。
2. 是否 2 分钟内能完成？是则建议 `do_now`。
3. 是否今天必须完成？是则建议 `daily`。
4. 是否有硬性时间？写入 Daily/Project，同时提示用户设置系统提醒。
5. 是否多步、有完成线、有交付物？是则进入 Project。
6. 是否属于已有项目？优先并入已有项目。
7. 是否没有完成线但需要长期维护？进入 Area。
8. 是否是方向、指标或阶段目标？进入 Goal/Plan。
9. 是否是外部资料？进入 Resources。
10. 是否是经验或异常？进入 Reviews。
11. 是否是稳定偏好/长期背景？进入 AI_Memory 候选。

## 三类主路径

从捕获到执行，最重要的只有三类：

| 主路径 | 含义 | 后续 |
| --- | --- | --- |
| 能立即执行 | 直接做或进 Daily | 完成后留 Daily，不长期维护 |
| 需要分步执行 | 进已有项目或新建项目 | 生成下一步，挂 Area/Goal |
| 要立即丢弃 | 删除或进入丢弃记录 | 不让系统变重 |

Resources、Reviews、AI_Memory 是非行动路径，只有在有沉淀价值时才进入。

## 分步项目的简化规则

分步事项优先问两个问题：

1. 它是否属于已有项目？
2. 如果不是，是否值得新建 Project Hub？

不要先问“属于哪个文件夹”。Project Hub 是分步执行的主线，Area 和 Goal 只是自动挂接的索引层。

```text
Inbox 条目 → existing_project / new_project → 生成 next_action → Daily 执行
```

## 新建项目的最低信息

新建项目只需要这些信息：

```md
---
type: project
status: active
area:
goal:
next_action:
review_cycle: weekly
---

# 项目名

## 完成标准

- [ ]

## 下一步

- [ ]
```

项目先跑起来，再逐步补资料。不要因为模板没填满而耽误执行。

## 建议表格式

Agent 必须先输出建议表：

| 编号 | 原文 | 路由 | 建议去向 | 下一步 | 理由 | 需确认 |
| --- | --- | --- | --- | --- | --- | --- |
| I001 | 服务器购买部署、同步、图床 | new_project | 30_Projects/个人基础设施与同步方案 | 列出需求和约束 | 多步、有交付物 | 是否新建项目 |

## 决策后执行格式

用户确认后，Agent 再输出并执行操作清单：

```json
{
  "approved_items": ["I001"],
  "operations": [
    {"type": "create_project_hub", "path": "30_Projects/个人基础设施与同步方案"},
    {"type": "append_daily_task", "task": "列出个人基础设施需求和约束"},
    {"type": "link_area", "area": "个人执行系统"}
  ]
}
```

## 判断例子

| 捕获 | 建议 | 原因 |
| --- | --- | --- |
| 买电动行李箱 | daily | 单步购买任务 |
| 服务器购买部署、Obsidian 同步、图床方案 | new_project | 多步、有技术决策和交付 |
| 南京卖房子 | new_project | 多步、高风险、需要资料和沟通 |
| 每周清理 Inbox | area_maintenance | 长期维护，无完成线 |
| 一篇 Obsidian 插件教程 | resource | 外部资料 |
| 今天因为调插件没执行第一任务 | review | 执行异常，有复盘价值 |
| 重复的临时链接 | discard | 无摘要、无关联 |

## 防止系统变重

- 没有下一步的项目不创建。
- 没有摘要的链接不长期保存。
- 没有稳定性的感受不进 AI_Memory。
- 不确定但低价值的内容优先丢弃。
- 每次路由后必须让 Daily 出现可以执行的下一步。
