# 执行中心原则

> 当前版本：V2.1 Project Execution Center  
> 这份文件是新版准则：任务管理系统必须围绕执行和项目展开。

## 一句话准则

系统的中心不是分类、资料、仪表盘或 AI 记忆，而是 **今日执行** 和 **项目推进**。

其他所有部分都只能服务于三件事：

1. 捕获：把输入接住。
2. 决策：看清项目、进度、风险和下一步。
3. 复盘：沉淀项目经验，改进后续执行。

## 系统重心

```text
10_Daily 今日执行
30_Projects 项目推进
```

这两个地方是执行现场。其余文件夹都是支撑层。

| 层级 | 文件夹 | 定位 |
| --- | --- | --- |
| 执行层 | `10_Daily` | 今天真正要做的 1-3 件事 |
| 执行层 | `30_Projects` | 所有分步事项和项目主线 |
| 捕获层 | `00_Inbox` | 临时输入，不承担长期管理 |
| 方向层 | `20_Goals` | 方向、指标、阶段验收 |
| 约束层 | `40_Areas` | 长期责任、SOP、维护规则、项目索引 |
| 资料层 | `50_Resources` | 支撑项目的资料、教程、素材 |
| 复盘层 | `60_Reviews` | 项目经验、异常、周/月复盘 |
| 展示层 | `70_Dashboards` | 项目状态、任务、目标和数据展示 |
| 记忆层 | `80_AI_Memory` | 给 AI 的稳定摘要 |
| 模板层 | `90_Templates` | 加速创建，不制造形式负担 |
| 历史层 | `99_Archive` | 完成项目和过期资料 |

## 判断规则

### 1. 能今天做完

进入 Daily。

```text
Inbox → Daily → 完成 → 留在当天 Daily
```

### 2. 不能今天做完，但有完成线

进入 Project。不要先纠结 Area、Goal、Resource。

```text
Inbox → Project → 下一步 → Daily
```

### 3. 没有完成线，但需要长期维护

进入 Area。Area 只放维护规则、SOP、周期任务和项目索引。

### 4. 没有行动价值

删除或丢弃记录。删除是保持执行系统轻盈的一部分。

## Project 优先原则

凡是需要分步推进的事情，第一归属是 Project。

Goal、Area、Resource、Review、AI_Memory 都不能替代 Project：

- Goal 提供方向和验收指标。
- Area 提供长期边界和维护规则。
- Resource 提供资料。
- Review 提供经验。
- AI_Memory 提供给 AI 的摘要。

它们不承担项目正文，不替代项目任务池。

## 仪表盘定位

仪表盘不是工作现场，而是决策视图。

每天默认看：

1. `10_Daily/当天日期.md`
2. [[../../70_Dashboards/项目推进台]]

有捕获需要处理时，再打开：

3. [[../../70_Dashboards/Inbox处理台]]

## Agent 的新版职责

Agent 不只是分类器，更应该是项目推进助手。

Agent 每次处理捕获内容时，优先回答：

1. 这是否需要进入项目？
2. 是否属于已有项目？
3. 如果新建项目，完成标准是什么？
4. 下一步行动是什么？
5. 需要挂哪个 Goal / Area 作为索引？

Agent 每次处理项目时，优先回答：

1. 当前项目卡在哪里？
2. 下一步是否足够小？
3. 是否有阻塞？
4. 是否应该进入今天 Daily？
5. 是否需要复盘、归档或沉淀？

## 防偏规则

- 不为了分类而分类。
- 不为了看板而看板。
- 不把 Inbox 当任务清单。
- 不把 Area 当项目正文。
- 不把 AI_Memory 当项目日志。
- 不把资料收集当项目推进。
- 不让任何项目没有下一步。
